# BowlRus Stats — контекст проекта

## Что это

Сайт со статистикой турниров по боулингу. Public: https://bowlrus.vercel.app

Владелец проекта — Александр (Александр не программист, владеет базовыми концепциями; на Python ориентируется лучше, чем на JS). Все объяснения — на русском. Тон — дружелюбный наставник, без снисходительности.

## Архитектура (мульти-спорт)

Сайт поддерживает несколько «спортов» — независимых соревнований со своими БД и страницами:

- **КЛБ** (Континентальная Лига Боулинга) — основной спорт, 4 сезона с 2023 г. БД MySQL `bowling`.
- **ЧР** (Чемпионат России) — отдельные ЧР по годам. БД MySQL `russian_championship_2026` для ЧР 2026.

В будущем могут добавиться: Кубок России, всероссийские соревнования.

Каждый спорт имеет свой SQLite-файл и свою папку JSON-витрин.

```
MySQL (локально)             SQLite (data/)        JSON (frontend/public/data/)   React (Vercel)
─────────────────            ─────────────────     ───────────────────────────    ──────────────
bowling                   →  klb.db             →  klb/...                     →  /klb/*
russian_championship_2026 →  chr2026.db         →  chr/...                     →  /chr/*
```

Backend (FastAPI) — мульти-спортный:
- `backend/db.py` — реестр `SPORTS = {"klb": ..., "chr": ...}` и `get_conn(sport)`. Новый спорт добавляется одной строкой.
- `backend/queries.py` — все SELECT'ы к витринам как чистые функции (`klb_*`, `chr_*`).
- `backend/main.py` — тонкие FastAPI-роуты по схеме `/api/{sport}/...`.

`scripts/export_to_json.py` использует те же query-функции, что и API — гарантия идентичности данных между API и статическими JSON.

**Ключевой принцип:** наружу попадает только то, что прошло через витрины и экспорт. Чувствительные поля остаются в MySQL и физически не покидают ноут.

### Генерируемые данные vs ручные ассеты

- **Генерируемые данные** — `frontend/public/data/{klb,chr}/`. Папка ПОЛНОСТЬЮ пересоздаётся экспортом (атомарная замена). Руками туда ничего не класть.
- **Ручные ассеты** (фото программ масла; в будущем фото игроков, эмблемы клубов) — `frontend/public/assets/klb/`. Экспорт эту папку НЕ трогает. Все будущие ассеты класть только сюда.
- На Windows перед `python scripts/export_to_json.py` остановить `npm run dev` — dev-сервер держит блокировку на `data/klb`, и атомарная замена падает с PermissionError (после того как rmtree уже снёс папку).

## Стек

- **Frontend:** Vite + React 18 + TypeScript + Tailwind CSS v3 + React Router v6 + Recharts.
- **Хостинг:** Vercel (Hobby, Personal).

- **Git:** ветка `main`, репо https://github.com/Narizinh0o/bowlrus.

## Структура БД КЛБ (`bowling`)

- **`tournaments`** — этапы Лиги: `tournament_id`, `name`, `year`, `season`, `stage`, `oil_pattern_id`, `oil_pattern_ptq_id`. 25 строк.
- **`players`** — игроки: `player_id`, `full_name`, `gender (М/Ж/U)`, `club_id`, `hand (R/L)`. 319 строк.
- **`clubs`** — клубы: `club_id`, `club_name`. 17 строк.
- **`teams`** — команды: `team_id`, `team_name`, `club_id`. 56 строк.
- **`games`** — отдельные игры: `game_id`, `tournament_id`, `player_id`, `team_id`, `match_id`, `stage_type`, `game_number`, `raw_score`, `hcp`, `event`. ~44 600 строк.
- **`matches`** — плей-офф матчи: `match_id`, `tournament_id`, `stage_type`, `player1/2_id`, `team1/2_id`, `score_str`, `rolloff_p1/p2`, `winner_id`, `winner_team_id`. ~4 660 строк.
- **`player_club_history`** — за какой клуб играл игрок в конкретном турнире (источник историчного клуба).
- **`player_team_history`** — за какую команду играл в конкретном турнире.
- **`oil_patterns`** — программы масла: `id`, `pattern_name`, `distance_ft`, `volume_ml`, `ratio`, `photo_file`. 24 строки. `ratio` может быть NULL (листы FLEX/IKON не содержат поле RATIO, в отличие от Kegel Library).

## Структура БД ЧР (`russian_championship_2026`)

3 таблицы — гораздо проще:

- **`players`** — `player_id`, `player_name`. 194 строки (после слияния дубликатов).
- **`games`** — `game_id`, `player_name`, `player_id`, `lane`, `game_number`, `play_date`, `time_start`, `time_end`, `total_score`, `event ('doubles' | 'doubles mix' | 'single')`. 2 100 строк, из них `single` — 6 черновых игр.
- **`frames`** — `game_id`, `frame_number`, `content`. ~21 000 строк.

**Фильтр на миграции:** в `migrate_chr.py` SQLite-копия исключает `event = 'single'` (черновик, у игр нет дорожки). На проде только `doubles` и `doubles mix`: 2 094 игры, 194 игрока. Если single дозаполнится — снять фильтр (константа `PUBLIC_EVENTS_SQL`).

## Зачёты (`event`)

КЛБ: `личный`, `командный`, `ветераны`. Все маленькими буквами. **Ветеранов отдельной страницей не показываем**, но в БД они есть и учитываются при разрешении проги масла (см. ниже).

ЧР: `doubles`, `doubles mix`. Фильтр игроков по зачёту через раздельные JSON: `chr/players.json` (агрегат), `chr/players_doubles.json`, `chr/players_doubles_mix.json`.

## Этапы КЛБ (`stage_type`)

Классификация для агрегатов:

- **Квалификация (personal):** `PTQ`, `Основная квалификация`
- **Квалификация (team):** `PTQ` (командный, редко)
- **Round Robin:** `RR`, `RR_A`, `RR_B`, `RR_C` — это одна стадия, разные группы. На UI всё, что начинается с `RR`, схлопывается в один пункт **«Round Robin»** (и в фильтре, и в разбивке). Сырьё не переписывать — нормализация только на уровне отображения (`displayStage(st) => st.startsWith('RR') ? 'Round Robin' : st`).
- **Плей-офф:** `1 этап`, `2 этап`, `Четвертьфинал`, `Полуфинал`, `Финал`, `Матч за 3 место`, `double elimination`, `Степледдер`
- **VPTQ** — ветеранский PTQ (event='ветераны'). Отдельной страницей не показываем, но в расчёте проги учитывается (играется на проге PTQ).

Регламент менялся каждый сезон, классификация — нормализация для отображения.

## ВАЖНО: «этап» — два разных понятия

- `tournaments.stage` — **порядковый номер турнира в сезоне** (грандфинал = 8, самый высокий). Это хронология.
- `games.stage_type` — **фаза внутри турнира** (квала, полуфинал, финал, RR…).

Не путать. Хронология турниров — по `(season, stage)`, НЕ по `tournament_id` (турниры заносились не по порядку: сезон 3 → id 1–15, сезоны 1–2 → id 16–30, сезон 4 → id 31+). Производное поле `seq = season*100 + stage`. «Последний клуб игрока» в свёрнутой строке = клуб из факта с максимальным `seq` (а не по max tournament_id — это была ошибка, исправлена).

## Программы масла (oil patterns)

- Связь с турнирами: `tournaments.oil_pattern_id` (основная прога, есть всегда) + `tournaments.oil_pattern_ptq_id` (прога PTQ, может быть NULL). Несколько турниров могут ссылаться на одну прогу (дедупликация по ссылке).
- Разрешение проги для стадии:
  `CASE WHEN stage_type IN ('PTQ','VPTQ') OR event='ветераны' THEN COALESCE(oil_pattern_ptq_id, oil_pattern_id) ELSE oil_pattern_id END`
- Фото отображается как иконка-ссылка (лайтбокс) на странице турнира; если PTQ-прога отличается от основной — две иконки. Путь в `patterns.json` → `/assets/klb/patterns/<file>`.

## Особенности счёта

- **Гандикап `hcp`:** у женщин +8 в личном/командном, у ветеранов индивидуально по возрасту (только ветеранский зачёт). **Все средние и лучшие игры считаются как `raw_score + hcp`** — «справедливый» итог.
- **Командные RR:** `raw_score` в `games` для `stage_type` IN ('RR','RR_A','RR_B','RR_C') командного зачёта — это **сумма двух игроков**. Для среднего делим на 2.
- **Командные плей-офф:** каждая запись — один игрок, `raw_score` не делим.

## Ролл-офф (`rolloff_p1`, `rolloff_p2`)

Дополнительные броски при ничейном счёте 1-1 в матче. Формат — строка символов:
- `0-9` — цифры (кегли). `X`/`Х` (латиница/кириллица!) — страйк = 10.
- `/` — **разделитель** «бросок в игре | бросок в матче», НЕ бросок. `\` — опечатка вместо `/`, тоже разделитель.
- Мусор (пробелы, `(...)`, `F` и пр.) — игнорируется.

Пример: `"96/X9"` = броски 9, 6, X, 9. Победитель ролл-офф — `matches.winner_id`.

## Витрины КЛБ (`migrate_klb.py`)

**Все средние — с гандикапом: `AVG(raw_score + hcp)`.**

### Готовые агрегаты (для списков и карточек)
- `v_player_personal` — личная статистика игрока: `player_id`, `player_name`, `gender`, `hand`, `current_club_name`, `games_total`, `avg_total`, `best_game`, `worst_game`, `tournaments_played`, разбивка квалы/RR/плей-офф.
- `v_team_stats` — командная: `team_id`, `team_name`, `club_*`, `games_total`, `avg_total`, `best_game`, разбивка, `latest_season`, `seasons_list`.
- `v_club_stats` — клубы: `players_count`, `teams_count`, `latest_season_with_team`.
- `v_player_rolloff` / `v_team_rolloff` — ролл-офф: `throws_count`, `throws_avg`, `matches_total`, `matches_won`, `winrate`.
- `v_tournament_personal` / `v_tournament_team` — результаты турнир × игрок/команда.

### Факт-витрины (для фильтруемых таблиц и разбивки)
Генерируются в `migrate_klb.py`, экспортируются в JSON:
- `player_facts.json` — зерно **игрок × турнир × стадия** (event='личный'). Поля строки: `g` (игры), `ss` (сумма raw+hcp), `bg`/`wg` (лучшая/худшая одиночная), `season`, `st` (stage_type), `patt` (разрешённый pattern_id), `club` (историчный, на момент турнира).
- `team_facts.json` — зерно **команда × турнир × стадия**: то же, `ss` = сумма team_score_adj (RR /2), `bg`/`wg` только из плей-офф.
- Справочники: `patterns.json`, `tournaments.json` (вкл. `seq`, main/ptq pattern), `players_lookup.json`, `teams_lookup.json`.

**Фронт агрегирует на клиенте под фильтрами:** `средний = Σss/Σg`, `лучшая = max(bg)`, `худшая = min(wg)`, `игр = Σg`, `турниров = distinct tid`. **Эту логику менять нельзя — она сверена с эталоном (Tableau).** Клубы выводятся группировкой игроков по `club` (отдельной факт-витрины клубов нет).

Фильтры на таблицах личного/командного: сезоны, турниры, стадии, программа (название + бегунки длина/объём/ратио, NULL-ratio проходят пока бегунок ратио не сужен), клуб, поиск по игрокам, пол, рука. Переключатель «Разбивка: нет / Сезон / Турнир» (меняется только ключ группировки).

## Витрина ЧР (`v_player_stats`)

См. `migrate_chr.py`. Считается из `frames`. Поля: `strike_attempts`, `strike_percent`, `spare_conversion_percent`, `single_pin_percent`. Префикс `S` в `content` нормализуется (удаляется) **только для расчёта агрегатов**.

## Фреймы ЧР и сплиты

`frames.content` хранит фрейм как строку: `"9/"`, `"X"`, `"S8/"` (сплит, закрыл). Парсер `parse_frame_content` в `backend/queries.py` → `{throws, splits}`:
- `throws[i]` — `"X"`, `"/"`, `"-"`, `"F"`, цифра.
- `splits[i]` — `true`, если после броска осталась сплит-расстановка.

Сплит возможен только по «свежим» 10 кеглям (первый бросок фрейма; 2-й/3-й бросок 10-го фрейма после страйка). Валидатор в `chr_game` ломает миграцию при противоречии (опечатка в MySQL).

Страница `/chr/games/:id` рисует фреймы как боулинг-скоринг: 10 ячеек, 10-й шире, сплит обведён кружком, фол красный, страйк/спэа янтарные. Накопительный счёт не показывается — только `total_score`.

## Как запускать локально

```powershell
# Backend (нужен только для регенерации JSON и отладки в /docs)
uvicorn backend.main:app --reload

# Frontend
cd frontend
npm run dev
```

Локально открывать сайт в обычном браузере по `http://127.0.0.1:5173/` (встроенный браузер VS Code и `localhost` через IPv6 могут капризничать; `127.0.0.1` надёжнее).

## Workflow обновления данных

1. Внести данные в MySQL Workbench.
2. **Остановить `npm run dev`** (Windows: иначе экспорт падает на блокировке `data/`).
3. Миграция: `python scripts/migrate_klb.py` (или `migrate_chr.py`) — пересоздаёт SQLite + витрины.
4. Экспорт: `python scripts/export_to_json.py` — JSON. Backend поднимать не обязательно.
5. `git add . && git commit -m "Update data" && git push` — Vercel передеплоит за 1-2 минуты.

## Темы и принципы

- **Тёмная тема** (Tailwind slate-900), акцент янтарный (amber). Шрифт **Inter**, числа выровнены вправо.
- Топ-3 в таблицах — медали 🥇🥈🥉 (на мобиле медальная колонка может скрываться).
- **Сайт адаптивный (desktop + mobile).** Целевая аудитория во многом мобильная.
- Всё на русском.
- **ID объектов передаём в JSON для роутинга, но НЕ показываем в UI.**
- Гандикаповый счёт отдельной колонкой не показываем — он уже включён в общий итог.

## Мобильная вёрстка

- **НЕ возвращать `min-width` на `body`** (в `index.css`) и нигде не задавать ширину больше ширины экрана — это рендерит весь сайт в десктопной ширине и ломает мобилку. `index.html` обязан содержать `<meta name="viewport" content="width=device-width, initial-scale=1.0">`.
- Адаптив — через брейкпоинт Tailwind (`md:`). При мобильных правках десктоп не менять (проверять регрессию).
- Таблицы личного/командного зачёта на мобиле: горизонтальный скролл внутри контейнера с `overflow-auto` + липкая шапка + липкий первый столбец; угловая ячейка липкая по двум осям с z-index выше шапки и тела; всем липким элементам — непрозрачный фон.
- Первый столбец на мобиле: мельче шрифт (`text-xs md:text-sm`), ограниченная ширина, имя «Фамилия Имя» в две строки с переносом длинных фамилий (только personal; команды не разбиваем).
- Карточки/ряды, которые на десктопе в строку (`flex`), на мобиле в столбик (`flex-col md:flex-row`).
- Навигация: десктопное меню `hidden md:flex`, гамбургер `md:hidden`. Фильтры на мобиле — bottom-sheet по кнопке «Фильтры».

## Безопасность

- `.env` всегда в `.gitignore`, пароли не коммитятся. SQLite-файлы (`data/*.db`) в `.gitignore`.
- MySQL слушает localhost. Репо public (можно переключить в private без последствий).
- В JSON попадают только данные, согласованные в этом файле.

## Что НЕ делать без явного запроса

- Не использовать `create-react-app` (только Vite). Не ставить Tailwind v4 (только v3).
- Не добавлять backend на Vercel как serverless (архитектура статическая).
- Не возвращать `min-width` / фиксированную ширину, ломающую мобильный layout.
- Не использовать яркие/розовые/неоновые цвета.
- Не делать парную статистику в ЧР (только индивидуальную).
- Не показывать ветеранов отдельной страницей в КЛБ (учитывать в расчётах, но не выводить как раздел).
- Не показывать ID объектов на сайте.
- Не показывать гандикаповый счёт отдельной колонкой.
- Не выгружать зачёт `single` в ЧР (черновик, фильтруется в `migrate_chr.py`).

## Принципы работы с кодом

Эти правила — для меня (Claude и Claude Code), когда я пишу или меняю код в этом репо.

### 1. Сначала подумать, потом писать
- Явно проговорить допущения. Если что-то неясно — остановиться и спросить, не угадывать.
- Если у задачи несколько разумных трактовок — показать их, не выбирать молча.
- Если есть способ проще — сказать, даже если уже начат сложный.

### 2. Простота
- Минимум кода, решающего задачу. Никакой «гибкости на будущее», которую не просили.
- Никаких абстракций ради одного использования. Никакой обработки невозможных сценариев.
- Если получилось 200 строк, а можно 50 — переписать.

### 3. Точечные правки
- Трогать только нужное. Не «улучшать» соседний код, форматирование, комментарии.
- Не рефакторить то, что не сломано. Подстраиваться под стиль файла.
- Мёртвый код рядом — сказать, не удалять самовольно.
- Каждая изменённая строка объясняется через исходный запрос.

### 4. Проверяемый результат
Перед многошаговой работой — короткий нумерованный план с критериями проверки на каждом шаге (не «сделать чтобы работало», а «после шага X должно появиться Y, возвращающее N строк»).

### 5. Решать проблемы, а не маскировать
1. Сначала диагностика (логи, запросы к БД, проверка данных). 2. Потом фикс.
**Заглушки и `try/except: pass` запрещены без явного согласования.** Если настоящее решение не выходит — остановиться и спросить: «вижу проблему X, корректно не выходит из-за Y, могу обойти костылём Z — окей?». Александр должен узнать о костыле ДО того, как он попадёт в репо.
